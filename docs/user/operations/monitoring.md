# Мониторинг очередей

## Основные инструменты

| Инструмент | Где открыть | Назначение |
|-----------|------------|-----------|
| **Фоновые задания** | Kafka / Администрирование / Фоновые задания | Состояние потоков, ошибки фоновых заданий |
| **Регламентное задание** | Kafka / Администрирование / Регламентное задание | Мониторинг выполнения регламентного задания |
| **РС «Исходящие сообщения»** | Kafka / Исходящие сообщения | Детальный просмотр очереди исходящих, ручной перезапуск, [очистка](queue-maintenance.md) |
| **РС «Входящие сообщения»** | Kafka / Входящие сообщения | Детальный просмотр очереди входящих, ручной перезапуск, [очистка](queue-maintenance.md) |

## История выгрузки объекта

Посмотреть, ушёл ли конкретный объект в Kafka, можно прямо из его формы — команда **«История выгрузки (Kafka)»** открывает форму списка РС «Исходящие сообщения» с отбором по записям этого объекта. Команда доступна двумя способами:

- **Подключаемая команда БСП** — появляется в командной панели форм всех объектов, участвующих в интеграции (группа «См. также»);
- **Глобальная команда с горячей клавишей ++ctrl+alt+h++** — определяет объект по активному окну и открывает его историю выгрузки; работает и из формы объекта, и из формы списка.

## Что смотреть каждый день

1. **Размер очередей** — нет ли постоянно растущего числа сообщений в статусах **Новое** / **Ошибка**.
2. **Состояние фоновых заданий** — все ли потоки работают.
3. **Регламентное задание** — не упало ли, выполняется ли по расписанию.
4. **Уведомления в Telegram** (если включён [контроль интеграции](../configuration/alerts.md)) — есть ли нарушения SLA.

## Что смотреть еженедельно

1. **Накопление ошибок** — сколько сообщений в статусах «Ошибка выгрузки» и «Ошибка обработки».
2. **Журнал регистрации 1С** — необработанные исключения в фоновых заданиях.
3. **Сроки хранения** сообщений — не разрастается ли размер регистров сверх разумного; см. [Обслуживание очередей](queue-maintenance.md).

## Автоматизированный мониторинг

Для продуктивного контура настройте:

- **[Алерты и контроль интеграции](../configuration/alerts.md)** — уведомления в Telegram при нарушении SLA.
- **[Внешнее логирование](../configuration/logging.md)** — выгрузка журнала обмена в ELK / Loki. В Kibana / Grafana можно построить дашборды.

## Что делать при росте очереди

Подробности — в [Типовые проблемы → Очередь растёт](troubleshooting.md#очередь-постоянно-растёт).

Краткая формула:

1. Увеличьте **подпотоки обработки** и **подпотоки транспорта** в диспетчере задач продюсера/консьюмера.
2. Включите **потоковый режим** регламентного задания.
3. Разделите нагрузку между **несколькими диспетчерами** или топиками.

## Примеры запросов для мониторинга

Готовые шаблоны запросов 1С, которые можно выполнять в консоли запросов, обработках-отчётах или регламентных проверках.

### Количество сообщений по статусам

```bsl
ВЫБРАТЬ
	Т.Статус КАК Статус,
	КОЛИЧЕСТВО(*) КАК Количество
ИЗ
	РегистрСведений.кфкИсходящиеСообщения КАК Т

СГРУППИРОВАТЬ ПО
	Т.Статус

УПОРЯДОЧИТЬ ПО
	Т.Статус
```

```bsl
ВЫБРАТЬ
	Т.Статус КАК Статус,
	КОЛИЧЕСТВО(*) КАК Количество
ИЗ
	РегистрСведений.кфкВходящиеСообщения КАК Т

СГРУППИРОВАТЬ ПО
	Т.Статус

УПОРЯДОЧИТЬ ПО
	Т.Статус
```

!!! tip "Вывод в Telegram / на почту"
    Комбинируйте эти запросы с регламентным заданием прикладной конфигурации и отправляйте сводку в чат/письмо. Для алертов по SLA отдельный механизм не нужен — он [встроен](../configuration/alerts.md).
